Performance hardening, API rate limits, and input validation across the stack. No new UI features — this release is entirely backend reliability and security work.

Performance

  • trade_audit_runs timestamp index + Redis cache — a DESC index on the timestamp column, plus a 60-second Redis cache on the latest audit run lookup, cuts repeated DB reads proportionally to polling frequency.

API Rate Limits

  • Mutation routes throttled — notes, calendar, and tags endpoints are now rate-limited (30–120 req/min depending on route), preventing unbounded DB writes from tight loops.

Input Validation

  • max_length on 12 schema fields — NoteBase, PostMortemBase, DailyJournalEntry, EntryChecklistItem, TradeBase currencies, and UserUpdate.display_name now enforce max_length constraints. Long strings return 422 instead of hitting MySQL strict mode.

Infrastructure

  • S3 backup integrity verification — the backup script now verifies gzip integrity after upload and fires a Slawk alert on failure.
  • change_audits retention policy — prune script and timestamp index added for efficient range-deletes on the audit log table.
  • Redis RDB persistence disabled — cache-only config in E2E/CI stack; no point snapshotting a pure cache.
  • MySQL slow query log enabled — via migration 039 for production visibility into slow queries.

Fixes

  • Superuser tag/signal/checklist scope — GET /tags, /signals, and /entry-checklist now return only the current user's own data (was returning all users' data for superusers).
  • Duplicate tag pill on quick-create — adding a tag that already exists no longer renders a duplicate pill in the new trade form.
